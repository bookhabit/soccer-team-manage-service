# 테스트 케이스(TC) 작성 가이드

이 문서는 FC Flow 프로젝트의 기능별 테스트 시나리오를 정의하기 위한 표준 가이드입니다.  
테스트 코드 작성이 아닌, **발생 가능한 모든 경우의 수(Edge Cases)를 도출하는 것**을 목적으로 합니다.  
정의된 케이스는 개발자가 **직접 수동 테스트**하는 기준으로 사용합니다.

---

## 1️⃣ 테스트 케이스 구성 원칙

모든 테스트 케이스는 **GIVEN(상황) - WHEN(행위) - THEN(결과)** 구조를 따르며,  
단순 기능 작동 여부뿐만 아니라 시스템 안정성을 확인하기 위한 예외 상황을 반드시 포함합니다.

### ✅ 검증 카테고리 정의

| 카테고리 | 설명 | 예시 |
|---|---|---|
| **01 (접근/권한)** | 인증되지 않은 사용자, 권한 없는 페이지 접근, 권한 변경 중간 상태 | 일반 멤버가 관리자 전용 버튼에 접근 |
| **02 (입력 유효성)** | 빈 값, 글자 수 초과, 잘못된 형식, 경계값 | 101자 메시지 입력 시 막힘 여부 |
| **03 (제출/요청)** | 중복 클릭, 네트워크 지연 중 재요청, 데이터 정합성 | 수락 버튼을 연속 두 번 누름 |
| **04 (성공)** | 정상 흐름에서의 데이터 반영 및 화면 전환 | 매칭 신청 성공 후 상태 PENDING 표시 |
| **05 (실패/예외)** | 서버 에러(500), 데이터 없음(404), 네트워크 끊김, 타임아웃 | 서버 점검 중 목록 조회 시 에러 뷰 노출 |

---

## 2️⃣ 테스트케이스 ID 규격

```
{기능명대문자}-{카테고리번호}-{순번}

예시:
  MPOST-02-003  → 매칭 게시글, 입력 유효성, 3번째 케이스
  MAPP-05-001   → 매칭 신청, 실패/예외, 1번째 케이스
  MFLOW-04-002  → 매칭 E2E 플로우, 성공, 2번째 케이스
```

---

## 3️⃣ 테스트 레이어별 정의 기준

| 레이어 | 정의 목적 | 주요 체크포인트 |
|---|---|---|
| **Unit** | 데이터 정합성 및 비즈니스 로직 검증 | 스키마 검증, 날짜 계산, 상태 파생 함수 |
| **Integration** | API 연동 및 데이터 흐름 검증 | API 응답 → 화면 반영, 쿼리 무효화 |
| **Component** | 유저 경험(UX) 및 인터랙션 검증 | 버튼 비활성화, 에러 메시지, 빈 상태 UI |
| **E2E** | 사용자 시나리오 완결성 검증 | 주요 플로우 처음~끝 성공 여부 |

---

## 4️⃣ 예외 및 엣지 케이스 체크리스트

케이스 생성 시 아래 시나리오를 반드시 포함한다.

- **빈 데이터 상태**: 목록이 없을 때 표시되는 Empty UI
- **네트워크 불안정**: 요청 중 연결이 끊겼을 때 재시도 또는 에러 알림
- **경계값 테스트**: 글자 수 제한의 마지막 글자(100자), 날짜 경계(경기 당일 등)
- **중복 요청**: 버튼을 연속 두 번 눌렀을 때 중복 처리 방지
- **동시성 이슈**: API 응답 전 페이지 이탈, 다른 사용자가 동시에 같은 작업 수행
- **권한 변경 중간 상태**: 관리자에서 일반 멤버로 변경된 직후 접근 시도
- **기기 환경**: 안드로이드 뒤로가기 버튼, 키보드가 입력창을 가리는 상황

---

## 5️⃣ 우선순위 기준

| 우선순위 | 기준 |
|---|---|
| **P1** | 핵심 비즈니스 로직 — 반드시 통과해야 출시 가능 |
| **P2** | 주요 예외 처리 — 사용자 경험에 직접 영향 |
| **P3** | 부가적인 UI 피드백, 엣지 케이스 |

---

## 6️⃣ 테스트케이스 문서 템플릿

```markdown
# {기능명} Test Cases

## 📌 테스트 요약
- **총 케이스 수:** N개
- **성공(04):** N개 / **실패 및 예외(01, 02, 05):** N개

## 1. Unit Test Cases (Logic & Schema)
| ID | GIVEN (상황/데이터) | WHEN (입력/실행) | THEN (기대 결과) | 우선순위 |
|---|---|---|---|---|

## 2. Integration Test Cases (API / 데이터 흐름)
| ID | GIVEN | WHEN | THEN | 우선순위 |
|---|---|---|---|---|

## 3. Component Test Cases (UI/UX 인터랙션)
| ID | GIVEN | WHEN | THEN | 우선순위 |
|---|---|---|---|---|

## 4. E2E Test Cases (핵심 사용자 플로우)
| ID | GIVEN | WHEN | THEN | 우선순위 |
|---|---|---|---|---|
```
