# @review — Planning Review Agent

기능명: **$ARGUMENTS**

`docs/plans/$ARGUMENTS.brief.md` 를 읽고 기획안의 완성도를 검토한다.
파일이 없으면 현재 대화에서 기획안 내용을 직접 받아 검토한다.

## 참조 문서
- `docs/01_concept.md` — 서비스 도메인 및 전체 맥락
- `server/prisma/schema.prisma` — 현재 데이터 모델

---

## 검토 항목

### 1. 비즈니스 로직 완결성
모든 규칙에 대해 누락된 케이스를 찾는다.

질문 패턴:
- "~한 경우" 가 명시됐다면 → "반대의 경우", "동시에 발생하는 경우" 는 어떻게 처리하는가?
- 숫자 제한이 있다면 → 정확히 그 한계값일 때는? (경계값)
- 권한 규칙이 있다면 → 권한이 변경되는 중간 상태는?

### 2. 엣지케이스 누락
아래 패턴을 기준으로 빠진 시나리오를 도출한다.

- **중복 요청**: 버튼을 연속으로 두 번 누르면?
- **동시성**: 두 사람이 동시에 같은 자원에 접근하면?
- **만료/취소**: 진행 중 상태가 취소되거나 만료되면?
- **권한 변경**: 작업 중 사용자의 역할이 바뀌면?
- **빈 상태**: 데이터가 0개일 때 UI는?
- **최대값**: 제한 수치를 정확히 초과하면?

### 3. UI/UX 요구사항 충돌 검사
- 비즈니스 로직과 UI 요구사항 간 모순이 있는가?
- 명시된 화면이 비즈니스 규칙을 모두 커버하는가?
- 로딩/에러/빈 상태 UI가 언급되어 있는가?

### 4. 데이터 모델 영향도
`server/prisma/schema.prisma` 를 읽고 판단한다.
- 신규 모델이 필요한가?
- 기존 모델에 필드 추가/변경이 필요한가?
- 관계(Relation) 변경이 다른 도메인에 영향을 주는가?

### 5. 보안 검토
- 인증(로그인) 없이 접근 가능한 엔드포인트가 생기는가?
- 권한(Role) 체크가 서버에서도 이루어지는가? (클라이언트 단독 제어 금지)
- 사용자 입력을 받는 곳에 Rate Limiting / 길이 제한이 명시됐는가?
- 코드·토큰 등 추측 가능한 값이 있다면 충분한 엔트로피가 보장되는가?

### 6. 성능 검토
- 리스트 조회가 있다면 페이지네이션 전략이 있는가?
- DB 조회 시 인덱스가 필요한 필드는 무엇인가?
- 타이머·폴링 등 주기적 UI 업데이트가 있다면 리렌더링 최적화 방안이 있는가?

### 7. 완료 기준(Definition of Done) 검토
- 각 사용자 시나리오에 대응하는 완료 기준이 존재하는가?
- 완료 기준이 테스트 가능한 형태로 작성됐는가?
